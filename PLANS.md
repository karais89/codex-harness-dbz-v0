# Codex 실행 계획 (ExecPlans)

이 파일은 이 저장소에서 ExecPlan을 작성하고 유지하는 방법을 정의합니다.

ExecPlan은 자기완결적인 실행 문서입니다. 새로운 Codex 세션이나 사람 개발자가 이전 채팅 기록에 의존하지 않고, 저장소와 ExecPlan만 읽고도 작업을 완료할 수 있어야 합니다.

ExecPlan은 살아 있는 문서입니다. 작업이 진행되거나, 새 사실을 발견하거나, 결정이 바뀔 때마다 갱신해야 합니다.

## 살아 있는 계획과 설계 결정

모든 ExecPlan은 필수 섹션을 포함하고 작업 중 계속 갱신해야 합니다.

프로젝트 상태, 현재 마일스톤, 활성 계획, 다음 단계, 알려진 제한 사항이 바뀌면 `docs/current-state.md`를 갱신합니다.

오래 유지해야 하는 프로젝트 결정이 생기면 ExecPlan의 `결정 기록`에 남기는 것과 별도로 `docs/decisions.md`를 갱신합니다.

ExecPlan의 `결정 기록`은 해당 계획 안에서 발생한 판단과 변경 이유를 기록합니다.

`docs/decisions.md`는 특정 계획을 넘어 계속 따라야 하는 프로젝트 결정을 기록합니다.

## ExecPlan 저장 위치와 파일 이름

ExecPlan은 저장소 루트의 `exec-plans/` 아래에 저장합니다.

파일 이름은 다음 형식을 사용합니다.

`NNN-short-kebab-title.md`

예시:

- `exec-plans/001-bootstrap.md`
- `exec-plans/002-first-playable-loop.md`
- `exec-plans/003-play-mode-verification.md`

## ExecPlan을 사용하는 경우

사소하지 않은 작업에는 ExecPlan을 사용합니다.

사소하지 않은 작업의 예시는 다음과 같습니다.

- 새로운 게임플레이 동작
- Unity 씬 변경
- 아키텍처 또는 폴더 구조 변경
- 워크플로우 변경
- 테스트 또는 검증 방식 변경
- 동작이나 구조를 바꾸는 리팩터링
- 패키지 추가 또는 제거
- 프로젝트 관례 변경

다음 작업에는 ExecPlan이 필요하지 않습니다.

- 오타 수정
- 작은 서식 수정
- 아주 작은 README 변경
- 동작이나 워크플로우에 영향을 주지 않는 주석

## 필수 읽기 순서

ExecPlan을 작성하거나 수정하기 전에 다음을 읽습니다.

1. `README.md`
2. `AGENTS.md`
3. `PLANS.md`
4. `docs/current-state.md`가 있으면 읽기
5. `docs/decisions.md`가 있으면 읽기
6. 관련 ExecPlan이 있으면 읽기

## 필수 섹션

모든 ExecPlan은 다음 섹션을 포함해야 합니다.

### 1. 목적

목표를 쉬운 말로 설명합니다.

계획이 완료된 뒤 지금은 할 수 없지만 새로 가능해지는 일을 설명합니다.

관찰 가능한 결과가 명확하지 않다면 “시스템 개선”이나 “코드 정리” 같은 모호한 목표는 피합니다.

### 2. 진행 상황

체크박스를 사용합니다.

작업이 시작되거나, 멈추거나, 방향이 바뀌거나, 완료될 때마다 이 섹션을 갱신합니다.

예시:

- [x] Unity 프로젝트 생성.
- [x] `README.md` 추가.
- [ ] Unity 프로젝트가 열리는지 확인.
- [ ] 최종 M0 상태를 GitHub에 push.

### 3. 맥락

프로젝트의 현재 상태를 설명합니다.

관련 파일, 폴더, 씬, 스크립트, 문서의 정확한 이름을 포함합니다.

독자가 이전 채팅 기록을 기억한다고 가정하지 않습니다.

### 4. 계획

작업을 작고 순서 있는 단계로 설명합니다.

각 단계는 Codex가 추측 없이 실행할 수 있을 만큼 구체적이어야 합니다.

넓은 수평적 준비 작업보다 작고 검증 가능한 수직 단위를 선호합니다.

### 5. 검증

작업 완료를 어떻게 증명할지 설명합니다.

검증은 관찰 가능해야 합니다.

다음 중 하나 이상을 사용합니다.

- Unity Editor 관찰
- Play Mode 동작
- Edit Mode 테스트
- 명령 출력
- 파일 존재 확인
- `git status` 확인
- 문서화된 수동 검증

코드나 파일이 변경되었다는 이유만으로 계획을 완료 처리하지 않습니다.

### 6. 결정 기록

중요한 결정은 다음 형식으로 기록합니다.

- 결정:
- 근거:
- 날짜:

범위가 바뀌거나, 트레이드오프가 발생하거나, 복잡한 방법 대신 더 단순한 방법을 선택한 경우 이 섹션을 사용합니다.

### 7. 예상 밖 발견

예상하지 못한 사실을 기록합니다.

예시:

- Unity가 예상과 다르게 파일을 생성했다.
- 패키지 또는 API를 사용할 수 없었다.
- 더 단순한 구현 경로를 발견했다.
- 원래 계획이 너무 컸다.

### 8. 회고

계획이 끝나면 다음을 요약합니다.

- 완료한 것
- 완료하지 못한 것
- 배운 것
- 다음에 해야 할 것
- 다음 계획을 시작할 준비가 되었는지 여부

## 실행 규칙

- 각 ExecPlan은 자기완결적으로 유지합니다.
- 이전 채팅 기록에 의존하지 않습니다.
- 범위는 검증 가능할 만큼 작게 유지합니다.
- 큰 준비 단계보다 수직 단위를 선호합니다.
- 작업이 바뀌면 진행 상황을 갱신합니다.
- 방향이 바뀌면 결정 기록을 갱신합니다.
- 실제 상황이 계획과 다르면 예상 밖 발견을 갱신합니다.
- 범위를 조용히 확장하지 않습니다.
- ExecPlan에 명시적으로 포함되고 정당화된 경우가 아니라면 MCP, 사용자 정의 skill, hook, subagent, 서드파티 패키지를 추가하지 않습니다.
- 공개 저장소 파일에 개인의 경력, 가족, 재정, 고용주 관련 민감 정보를 포함하지 않습니다.

## 완료 기준

ExecPlan은 다음을 모두 만족할 때만 완료됩니다.

1. 요청한 결과가 존재합니다.
2. 결과가 검증되었습니다.
3. 진행 상황 섹션이 완전히 갱신되었습니다.
4. 결정 기록에 주요 선택이 반영되었습니다.
5. 회고가 작성되었습니다.
6. 프로젝트 상태가 바뀌었다면 `docs/current-state.md` 또는 현재 프로젝트 상태 문서를 갱신했습니다.
7. 지속적으로 따라야 할 결정이 생겼다면 `docs/decisions.md` 또는 현재 결정 기록 문서를 갱신했습니다.
8. commit 후 `git status`가 clean이거나, 남은 변경 사항을 명시했습니다.

## Unity 전용 검증

Unity 작업의 검증에는 다음 중 무엇을 확인했는지 명시해야 합니다.

- Unity 프로젝트가 컴파일 오류 없이 열린다.
- 필요한 씬이 존재한다.
- Play Mode 동작을 수동으로 확인했다.
- 관련 스크립트가 컴파일된다.
- 테스트가 있다면 통과한다.
- Console 오류를 확인했다.
- 문서화된 절차를 따라 결과를 재현할 수 있다.

## 계획 크기 기준

ExecPlan에 관련 없는 목표가 너무 많다면 분리합니다.

좋은 예:

- Unity 프로젝트와 최소 Codex 하네스 부트스트랩.
- 첫 플레이 가능 루프 구현.
- 턴 제한 실패 상태 추가.

너무 넓은 예:

- 전체 게임 만들기.
- 모든 아키텍처와 게임플레이 설정.
- 완전한 AI 워크플로우 시스템 추가.

## 계획 변경

실행 중 계획이 바뀌면 다음을 수행합니다.

1. 계획 섹션을 갱신합니다.
2. 결정 기록 항목을 추가합니다.
3. 진행 상황을 갱신합니다.
4. 새 정보 때문에 변경했다면 그 이유를 예상 밖 발견에 설명합니다.

오래된 계획을 조용히 계속 따르지 않습니다.
