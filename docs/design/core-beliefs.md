# Core Beliefs

상태: 승인됨

## 목적

이 문서는 Delivery Bot Zero가 유지해야 하는 가장 상위의 design 기준과 워크플로우 기준을 정리한다.

Delivery Bot Zero는 작은 Unity 2D 퍼즐 게임이면서, Codex 기반 Unity 개발 워크플로우를 검증하기 위한 실험 프로젝트다. 이 문서는 플레이어가 무엇을 경험해야 하는지와 Codex가 어떤 기준을 넘지 말아야 하는지를 함께 정의한다.

이 문서는 2026-05-19에 사용자가 명시적으로 승인했다. Codex는 이 문서를 Delivery Bot Zero의 상위 design baseline으로 사용할 수 있다.

## 핵심 원칙

- Delivery Bot Zero는 작은 배달 로봇을 조작해 짧고 명확한 2D 퍼즐 문제를 풀고, 물건을 목적지까지 전달하는 게임이다.
- 플레이어는 차분하게 상황을 읽고, 부담 없이 다시 시도하며, 해법을 발견했을 때 작게 기분 좋은 성취감을 느껴야 한다.
- 초기 플레이어 경험은 짧고 읽기 쉬운 배달 퍼즐 루프가 플레이어에게 명확히 전달되는지 검증하는 데 집중한다.
- Codex는 player-facing gameplay 결정을 사용자 승인 없이 확정하지 않는다. Codex는 선택지를 제안할 수 있지만, 핵심 gameplay 결정은 사용자의 명시 승인을 받아야 한다.
- 프로젝트 완료 기준은 코드 생성 자체가 아니다. Unity Play Mode 검증, 검증 결과 기록, 다음 세션에서 이어받을 수 있는 handoff 가능 상태가 함께 갖춰져야 한다.

## 초기 마일스톤에서 하지 않을 것

- 속도 경쟁을 다루지 않는다.
- 복잡한 물리 퍼즐을 다루지 않는다.
- 적 또는 위협 요소를 다루지 않는다.
- 점수 최적화를 다루지 않는다.
- 긴 레벨을 다루지 않는다.
- 서사 연출을 다루지 않는다.

초기 마일스톤에서는 먼저 짧고 읽기 쉬운 배달 퍼즐 루프가 플레이어에게 명확히 전달되는지 검증한다.

## 성공과 실패 판단 기준

초기 플레이어 경험은 플레이어가 최소한의 설명만으로 시작 상태, 목표, 조작 결과, 배달 성공 여부, 재시도 가능성을 이해하고, 한두 번의 시행착오 뒤 스스로 해법을 발견할 수 있으면 성공으로 본다.

반대로 플레이어가 목표, 현재 상태, 조작 결과, 배달 성공 여부, 실패 이유를 이해하지 못하거나, 재시도가 귀찮게 느껴지거나, 해법을 찾기 전에 조작 또는 화면 해석에서 막힌다면 실패 신호로 본다.

## Codex가 임의로 확정하지 않을 것

Codex는 다음 결정을 사용자 승인 없이 확정하지 않는다.

- 핵심 조작 방식.
- 이동 규칙.
- pickup 또는 deliver 규칙.
- 실패 또는 재시도 규칙.
- 새로운 퍼즐 요소.
- 레벨 목표.
- 점수, 시간, 위협 요소 도입.

## 하네스 확장 원칙

007 이후 M1 완료 전까지는 hooks, MCP, custom skills, subagents, 별도 template 폴더 같은 하네스 확장을 하지 않는다.

단, M1 진행 중 실제 실패가 반복되어 진행이 막힐 때만 하네스 보강을 검토한다. 보강 검토가 필요해지면 먼저 실패 사례와 막힌 지점을 기록하고, 사용자와 범위를 확인한 뒤 별도 절차로 다룬다.

## 아직 정하지 않을 것

이 문서는 다음 세부 사항을 확정하지 않는다.

- `짧고 읽기 쉬움`의 실제 길이.
- 초기 난이도.
- 화면 구성.
- M1 First Playable Loop의 세부 규칙.
- 이동, pickup, deliver, result, retry의 구체 규칙.
- 좌표, 수치, 씬 구조.
- C# 스크립트 이름.
- 구현 방식.

위 항목은 후속 gameplay design 문서와 ExecPlan에서 사용자 승인을 거쳐 다룬다. `core-beliefs.md`만으로는 M1 gameplay ExecPlan을 시작할 수 없다.
