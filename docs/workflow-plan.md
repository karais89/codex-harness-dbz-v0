# 워크플로우 규칙 개선 초안

상태: 초안

목적: M1을 다시 시작하기 전에 ExecPlan 생성 전 사용자 의도 확인과 구현 승인 게이트를 정의한다.

## 배경

M1 진행 중 사용자 합의 없이 다음 산출물이 만들어지고 구현까지 시작되었다.

- `docs/game-brief.md`
- `exec-plans/004-first-playable-loop.md`
- `Assets/Scripts/DeliveryBotZero/FirstPlayableLoop.cs`

이 흐름은 재현성은 있었지만, 사용자가 생각한 M1 방향과 Codex가 가정한 방향이 달랐을 때 혼란을 만들었다.

문제의 핵심은 ExecPlan 자체가 아니라, ExecPlan을 만들기 전에 사용자 의도를 충분히 확인하는 단계가 없었다는 점이다.

## 목표 흐름

앞으로 M1 같은 사소하지 않은 작업은 다음 순서로 진행한다.

```text
논의
-> 범위 합의
-> 명시적 ExecPlan 작성 요청
-> 구현 승인 대기 상태의 ExecPlan 생성
-> 명시적 구현 승인
-> 구현
-> 검증과 문서 갱신
```

이 흐름의 목적은 사용자가 처음부터 완성된 프롬프트를 작성해야 하는 부담을 줄이고, Codex가 부족한 정보를 질문으로 끌어내게 하는 것이다.

## 요청 분류 규칙

Codex는 사용자 요청을 먼저 분류한다.

- 분석, 논의, 검토 요청이면 파일 변경, ExecPlan 생성, 구현을 하지 않는다.
- 오타 수정, 작은 문서 보정처럼 사용자가 명확히 요청한 작은 변경은 ExecPlan 없이 처리할 수 있다.
- 게임플레이, Unity 씬, 워크플로우, 구조 변경은 사소하지 않은 작업으로 보고 ExecPlan 흐름을 따른다.

## ExecPlan 전 인터뷰 규칙

ExecPlan을 만들기 전에 Codex는 다음 항목이 충분한지 판단한다.

- 목표
- 성공 기준
- 포함 범위
- 제외 범위
- 제약
- 검증 방법
- 현재 상태와 영향 범위

부족한 정보가 있으면 바로 계획을 만들지 않고 질문한다.

질문 방식은 다음 원칙을 따른다.

- 코드나 문서에서 확인할 수 있는 사실은 먼저 직접 확인한다.
- 질문은 한 번에 하나의 중요한 불확실성만 다룬다.
- 질문마다 현재 이해, 막힌 결정, 추천 답안을 함께 제시한다.
- 답변을 받으면 결정된 내용을 짧게 갱신한다.
- 중요한 불확실성이 남았을 때만 다음 질문을 한다.

인터뷰 톤은 부드러운 요구사항 수집보다 `grill-me`에 가깝게 둔다. 즉, 계획의 약한 가정과 빠진 판단을 적극적으로 확인한다.

## ExecPlan 생성 게이트

Codex는 사용자가 범위에 동의했다는 이유만으로 ExecPlan 파일을 만들지 않는다.

ExecPlan 파일은 사용자가 다음처럼 명확히 요청했을 때 만든다.

- `이 범위로 ExecPlan 작성해`
- `계획 작성해`
- `이 내용으로 실행 계획 만들어줘`

`좋아`, `맞아`, `괜찮아` 같은 동의 표현만으로는 ExecPlan 생성 요청으로 보지 않는다.

## 범위 기록 방식

별도 `game-brief` 또는 `game-spec` 문서는 기본 산출물로 만들지 않는다.

사용자와 합의한 범위는 ExecPlan 안에 `사용자 승인 범위` 항목으로 기록한다.

별도 범위 문서가 필요한 경우에는 다음 조건을 만족해야 한다.

- 사용자가 명시적으로 요청했다.
- 또는 ExecPlan 하나에 넣기에는 범위 설명이 지나치게 길다.
- 또는 여러 ExecPlan이 같은 범위를 공유해야 한다.

이 경우에도 문서 상태를 `초안`, `승인 대기`, `승인됨` 중 하나로 명시한다.

## 구현 승인 게이트

ExecPlan 생성과 구현은 분리한다.

새 ExecPlan은 기본적으로 `구현 승인 대기` 상태로 시작한다.

구현은 사용자가 파일 변경 의도가 분명한 표현을 했을 때만 시작한다.

구현 승인으로 보는 예:

- `이 계획대로 구현해`
- `진행해`
- `적용해`
- `수정해`
- `커밋해`

구현 승인으로 보지 않는 예:

- `좋아`
- `맞아`
- `괜찮아`
- `일단 봐줘`
- `논의해보자`

## 작은 변경 예외

다음 작업은 사용자가 명확히 요청했다면 ExecPlan 없이 바로 처리할 수 있다.

- 오타 수정
- 작은 서식 수정
- 작은 README 또는 문서 문구 보정
- 현재 상태 문서의 단순 보정
- 동작이나 워크플로우에 영향을 주지 않는 주석

단, 작은 변경이라도 프로젝트 상태, 다음 단계, 알려진 제한 사항이 바뀌면 `docs/current-state.md`를 갱신한다.

## 참고한 인터뷰 원칙

다음 외부 skill은 설치하거나 저장소에 추가하지 않는다. 현재 프로젝트는 아직 사용자 정의 skill을 도입하지 않는다는 원칙을 유지한다.

다만 인터뷰 방식의 참고 원칙으로 사용한다.

- `deep-interview`: 모호한 요청을 바로 실행하지 않고 목표, 범위, 제약, 완료 기준을 질문으로 명확히 한다.
- `grill-me`: 계획이나 설계의 약한 지점을 질문으로 압박해 공유된 이해에 도달한다.

참고 링크:

- https://github.com/devbrother2024/skills/blob/main/deep-interview/SKILL.md
- https://github.com/mattpocock/skills/blob/main/skills/productivity/grill-me/SKILL.md

## 검증 기준

이 초안 문서는 다음을 만족하면 작성 완료로 본다.

- `docs/workflow-plan.md`가 존재한다.
- 문서가 한글로 작성되어 있다.
- 상태가 `초안`으로 명시되어 있다.
- ExecPlan 전 인터뷰 규칙이 들어 있다.
- ExecPlan 생성과 구현 승인 게이트가 분리되어 있다.
- 별도 `game-brief` 문서를 기본 생성하지 않는다는 규칙이 들어 있다.
- 외부 skill은 설치하지 않고 참고 원칙으로만 사용한다고 명시되어 있다.
