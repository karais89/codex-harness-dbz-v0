# 워크플로우 규칙 개선 초안

상태: 초안

목적: M1 같은 사소하지 않은 작업을 시작하기 전에 사용자 의도 확인, 공유된 이해 검증, ExecPlan 생성, 구현 승인 게이트를 정의한다.

## 배경

M1 진행 중 사용자 합의 없이 다음 산출물이 만들어지고 구현까지 시작되었다.

- `docs/game-brief.md`
- `exec-plans/004-first-playable-loop.md`
- `Assets/Scripts/DeliveryBotZero/FirstPlayableLoop.cs`

이 흐름은 재현성은 있었지만, 사용자가 생각한 M1 방향과 Codex가 가정한 방향이 달랐을 때 혼란을 만들었다.

문제의 핵심은 ExecPlan 자체가 아니라, ExecPlan을 만들기 전에 사용자 의도를 충분히 확인하고 공유된 이해에 도달했는지 검증하는 단계가 부족했다는 점이다.

## 목표 흐름

앞으로 M1 같은 사소하지 않은 작업은 다음 순서로 진행한다.

```text
논의
-> grill-me 방식 인터뷰
-> 공유된 이해 검증
-> ExecPlan 생성 여부 확인
-> ExecPlan 생성
-> Codex 자체 리뷰
-> 사용자 검토용 요약 제공
-> 구현 승인 확인
-> 구현
-> 검증과 문서 갱신
```

이 흐름의 목적은 사용자가 처음부터 완성된 프롬프트를 작성해야 하는 부담을 줄이고, Codex가 부족한 정보를 질문으로 끌어내게 하는 것이다.

## 요청 분류 규칙

Codex는 사용자 요청을 먼저 분류한다.

- 분석, 논의, 검토 요청이면 파일 변경, ExecPlan 생성, 구현을 하지 않는다.
- 오타 수정, 작은 문서 보정처럼 사용자가 명확히 요청한 작은 변경은 ExecPlan 없이 처리할 수 있다.
- 게임플레이, Unity 씬, 워크플로우, 구조 변경은 사소하지 않은 작업으로 보고 ExecPlan 흐름을 따른다.

## ExecPlan 전 인터뷰 규칙

ExecPlan 전 인터뷰는 `grill-me` 방식을 기본으로 한다.

Codex는 사용자의 요청이 짧거나 불완전해도 바로 ExecPlan을 만들지 않는다. 공유된 이해에 도달할 때까지 계획 또는 설계의 중요한 결정 분기를 하나씩 질문한다.

질문 방식은 다음 원칙을 따른다.

- 질문은 한 번에 하나만 한다.
- 각 질문에는 Codex의 추천 답안을 포함한다.
- 코드나 문서에서 확인할 수 있는 사실은 사용자에게 묻지 않고 직접 확인한다.
- 사용자의 요청이 충분히 구체적이면 질문 수를 줄이고, 남은 모호성이나 되돌리기 비싼 결정만 확인한다.

이 인터뷰의 목적은 질문을 많이 하는 것이 아니라, 사용자와 Codex가 같은 작업을 상상하고 있는지 검증하는 것이다.

## 공유된 이해 검증

Codex가 인터뷰가 충분히 진행되었다고 판단하면 바로 종료하지 않고 공유된 이해 검증 단계를 수행한다.

공유된 이해 검증에서는 다음을 요약한다.

- 목표
- 성공 기준
- 포함 범위
- 제외 범위
- 제약
- 검증 방법
- 위험한 가정
- 남은 열린 질문
- 되돌리기 비싼 결정

Codex는 요약을 제시한 뒤 사용자에게 이 내용이 의도와 맞는지 확인한다.

사용자가 공통된 의견에 도달했다고 확인하면 인터뷰를 종료한다.

사용자가 틀리거나 빠진 부분을 지적하면, 해당 항목에 대해 다시 `grill-me` 방식으로 질문하고 요약을 갱신한다.

## 검토 보조 산출물

공유된 이해 검증 요약은 기본적으로 채팅으로 제공한다.

계획이 복잡하거나 사용자가 요청한 경우에만 Codex는 검토 보조 산출물을 제안할 수 있다. 제안 후 사용자가 동의한 경우에만 생성한다.

허용되는 검토 보조 산출물은 다음과 같다.

- ExecPlan 요약 문서
- 검토용 HTML

이 산출물은 구현 산출물이 아니라 사용자 검토를 돕기 위한 보조 자료로 표시한다.

별도 `game-brief` 또는 `game-spec` 문서는 기본 산출물로 만들지 않는다.

## ExecPlan 생성 게이트

Codex는 인터뷰와 공유된 이해 검증이 끝난 뒤, 사용자에게 다음처럼 묻는다.

```text
이 내용으로 ExecPlan을 만들까요?
```

이 질문 직후에는 사용자의 짧은 긍정 답변도 ExecPlan 생성 승인으로 본다.

예:

- `네`
- `좋아`
- `진행해`
- `만들어줘`
- `작성해줘`

단, 공유된 이해 검증 없이 나온 일반적인 `좋아`, `괜찮아` 같은 동의 표현은 ExecPlan 생성 승인으로 보지 않는다.

새 ExecPlan은 기본적으로 `구현 승인 대기` 상태로 시작한다.

## ExecPlan 생성 후 자체 리뷰

ExecPlan을 생성한 직후 Codex는 사용자가 검토하기 전에 자체 리뷰를 수행한다.

자체 리뷰에서는 다음을 확인한다.

- 인터뷰 합의와 ExecPlan 내용이 일치하는가
- 목표, 범위, 제외 범위, 검증 방법이 빠지지 않았는가
- 구현 승인 전인데 구현 작업이 포함되어 있지 않은가
- 범위가 조용히 확장되지 않았는가
- 위험한 가정이나 열린 질문이 표시되어 있는가
- 완료 기준이 관찰 가능한가

자체 리뷰 결과는 기본적으로 채팅에 요약한다.

문제가 발견되면 Codex가 ExecPlan을 수정한 뒤 다시 자체 리뷰한다.

문제가 없으면 사용자 검토용 요약을 제공한다.

## 사용자 검토용 요약

ExecPlan 생성과 자체 리뷰가 끝나면 Codex는 사용자 검토용 요약을 제공한다.

요약에는 다음을 포함한다.

- 목표
- 포함 범위
- 제외 범위
- 검증 방법
- 현재 상태: 구현 승인 대기
- 사용자가 특히 확인해야 할 결정 3개 이하

요약은 기본적으로 채팅으로 제공한다.

복잡한 계획이거나 사용자가 요청하면 검토용 문서 또는 HTML을 제안할 수 있다.

## 구현 승인 게이트

ExecPlan 생성, 자체 리뷰, 사용자 검토용 요약까지 끝난 뒤 Codex는 사용자에게 다음처럼 묻는다.

```text
이 ExecPlan대로 구현을 시작할까요?
```

이 질문 직후에는 사용자의 짧은 긍정 답변도 구현 승인으로 본다.

예:

- `네`
- `진행해`
- `구현해`
- `적용해`

단, 검토 중 나온 일반적인 `좋아`, `맞아`, `괜찮아` 같은 동의 표현은 구현 승인으로 보지 않는다.

## 작은 변경 예외

다음 작업은 사용자가 명확히 요청했다면 ExecPlan 없이 바로 처리할 수 있다.

- 오타 수정
- 작은 서식 수정
- 작은 README 또는 문서 문구 보정
- 현재 상태 문서의 단순 보정
- 동작이나 워크플로우에 영향을 주지 않는 주석

단, 작은 변경이라도 프로젝트 상태, 다음 단계, 알려진 제한 사항이 바뀌면 `docs/current-state.md`를 갱신한다.

## 참고한 인터뷰 원칙

다음 외부 skill은 설치하거나 저장소에 추가하지 않는다. 현재 프로젝트는 아직 사용자 정의 skill을 도입하지 않는다는 원칙을 유지한다.

다만 인터뷰 방식의 참고 원칙으로 사용한다.

- `grill-me`: 기본 인터뷰 방식으로 참고한다.
- `deep-interview`: 질문 형식과 종료 요약의 보조 참고로만 사용한다.
- `brainstorming`: 복잡한 계획의 검토 보조 산출물 아이디어로만 참고한다.

참고 링크:

- https://github.com/devbrother2024/skills/blob/main/deep-interview/SKILL.md
- https://github.com/mattpocock/skills/blob/main/skills/productivity/grill-me/SKILL.md
- https://github.com/obra/superpowers/blob/main/skills/brainstorming/SKILL.md

## 검증 기준

이 초안 문서는 다음을 만족하면 작성 완료로 본다.

- `docs/workflow-plan.md`가 존재한다.
- 문서가 한글로 작성되어 있다.
- 상태가 `초안`으로 명시되어 있다.
- ExecPlan 전 인터뷰가 `grill-me` 기반으로 정의되어 있다.
- 공유된 이해 검증 단계가 들어 있다.
- ExecPlan 생성과 구현 승인 게이트가 분리되어 있다.
- ExecPlan 생성 후 자체 리뷰 단계가 들어 있다.
- 사용자 검토용 요약 규칙이 들어 있다.
- 별도 `game-brief` 문서를 기본 생성하지 않는다는 규칙이 들어 있다.
- 외부 skill은 설치하지 않고 참고 원칙으로만 사용한다고 명시되어 있다.
