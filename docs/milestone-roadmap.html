M1 핵심 루프

첫 구현 목표는 기능을 키우는 것이 아니라, 가장 작은 시작-결과-재시작 흐름을 완성하는 것이다.

Start Move Pickup Deliver Result Retry

마일스톤별 기준

각 단계는 하나의 핵심 질문에 답해야 하며, 다음 단계로 넘어가기 전에 검증 기록을 남긴다.

M0

프로젝트 부트스트랩

Codex가 일할 최소 작업장이 준비됐는가?
목표
  • Unity 프로젝트, GitHub repo, README, AGENTS, PLANS, docs, exec-plans 구조를 준비한다.
  • Codex가 repo 목적과 다음 작업을 설명할 수 있게 한다.

증명: Codex가 일할 수 있는 최소 작업장이 준비됐다.

M1

첫 플레이어블 루프

작은 게임 루프를 실제로 만들 수 있는가?
범위
  • 로봇 시작 위치, 한 칸 이동, Pickup, Delivery, Turn Limit, Clear/Failed, Retry를 구현한다.
  • Unity Play Mode에서 1분 안에 클리어, 실패, 재시작을 확인한다.

증명: Codex가 Unity에서 작은 플레이 루프를 만들었다.

M2

Codex 변경 작업 검증

기존 기능을 망가뜨리지 않고 수정 가능한가?
추천 변경
  • 장애물 Tile 추가를 우선 후보로 둔다.
  • 기존 Pickup, Deliver, Retry 흐름이 깨지지 않아야 한다.
  • 장애물에 막히는 동작을 Play Mode에서 확인한다.

증명: 최초 생성 이후의 두 번째 수정도 안전하게 수행했다.

M3

검증/테스트 루프 정착

감이 아니라 반복 기준으로 검증 가능한가?
검증 방식
  • 수동 Play Mode 체크리스트를 만든다.
  • 가능하면 간단한 Edit Mode Test 1~3개를 추가한다.
  • 최소 1회 이상 전체 체크리스트를 통과한다.

증명: Codex 작업 후 무엇을 확인해야 하는지 고정됐다.

M4

새 세션 핸드오프 검증

repo 문서만 보고 새 Codex 세션이 이어받을 수 있는가?
성공 기준
  • 프로젝트 목적, 완료 상태, 현재 마일스톤, 다음 작업을 정확히 요약한다.
  • 다음 작업을 과하게 확장하지 않는다.
  • 사용자가 긴 설명을 다시 하지 않아도 된다.

증명: repo 기반 기억과 문서화가 실제로 작동한다.

M5

공개 가능한 포트폴리오 패키징

실험 결과를 외부에 보여줄 증거물로 정리할 수 있는가?
산출물
  • README에 게임 설명과 AI 개발 워크플로우를 정리한다.
  • 30~60초 플레이 영상 또는 GIF, 주요 스크린샷, 회고 문서를 준비한다.
  • Codex 작업과 사람의 검증을 분리해 기록한다.

증명: 개인 학습을 공개 가능한 실험 산출물로 전환했다.
